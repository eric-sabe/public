Roles and model tiers

Two model tiers, ten roles, one escalation rule

The transferable idea is not the cast. It is the operating model: every role has a default tier and the decision to spend a stronger model is a written rule, not a judgment call made fresh each time.

The rule. Judgment work (product, architecture, security, testing, planning, model-optimization) defaults to the strong tier. Execution work (implementation, bug investigation, infrastructure, docs) defaults to the cheap tier and escalates to the strong tier when the task involves architectural decisions, cross-cutting refactors, high ambiguity, concurrency or data migrations, performance- or security-critical paths, or synthesis of conflicting sources. Routing to the right role matters more than the tier: the wrong role burns context on any model.

Review and scanning

How we actually use CodeRabbit and the security scanners

Two distinct jobs. CodeRabbit reads the diff for the bugs that compile, lint clean, and pass shallow tests. The scanners watch for secrets, vulnerable dependencies, and container CVEs. Both feed the same downstream record.

CodeRabbit the semantic reviewer

  • Cloud auto-review is off, on purpose. .coderabbit.yaml sets auto_review.enabled: false so we do not burn CI on a post-push review loop. The review happens before the push, so the PR opens already clean.
  • Local CLI against origin. cr review --agent --base origin/main produces findings the agent acts on. We use origin/main because local main is often stale in a worktree.
  • Triage and cap. Fix Critical and Warning, leave Info to discretion, re-run once to confirm clean, then stop. The loop is capped at about two passes so an agent never grinds.
  • Findings are persisted. After the PR opens, the findings go into one comment marked <!-- cr-cli-findings -->: each finding and how it was resolved. With no cloud threads, this comment is the durable record.
  • Deeper pass on demand. /code-review ultra runs a multi-agent cloud review of the branch or a PR when a change warrants more than the local pass.
That findings comment is not paperwork. It is the sensor for the learning loop below: closeout greps the marker across every PR in the project.

Scanners secrets, supply chain, CVEs

  • Secret scanning, two stages. A pre-commit gitleaks hook scans staged changes for fast local feedback; a non-bypassable CI job (secret-scan.yml) rescans the event's commit range as the backstop. A narrow allowlist lives in .gitleaks.toml; a real secret is rotated, never allowlisted.
  • Container CVE gate. On main builds, Trivy scans each image for Critical and High vulnerabilities with --ignore-unfixed and fails the build on a hit.
  • Results land in GitHub Advanced Security. Scanner output is uploaded as SARIF through github/codeql-action/upload-sarif, so findings surface in the GitHub code-scanning dashboard.
  • Dependency updates. Dependabot runs weekly for npm, Actions, and Docker. Minor and patch bumps are grouped; majors get individual PRs; its own PRs get a cheap quick-check instead of the full matrix.
Honest scope note: there is no standalone CodeQL analysis job today. The semantic-bug coverage CodeQL would give is provided by the CodeRabbit pass plus the integration and concurrency test suites; the scanning layer above is gitleaks, Trivy, and Dependabot feeding the same code-scanning surface.
Design

The design system is a skill, and fidelity is a loop

Two ideas worth copying. First, the brand is not tribal knowledge re-explained each session; it is a loadable capability. Second, the gap between a design proposal and the running app is closed by an agent working in a real browser, not by reading screenshots.

  • Built in Claude, embedded as a skill. We designed the system in Claude, then captured it in the repo at .claude/skills/. Any agent building UI, production or throwaway prototype, loads it and inherits the brand instead of re-deriving it.
  • Tokens as the source of truth. One file, colors_and_type.css: a teal primary, slate neutrals, paired semantic and sentiment palettes, a seven-stop chart palette, a 12-to-48px type scale on Inter and JetBrains Mono, a 4px spacing base, plus radii, shadows, motion, and layout tokens. Dark mode is canonical.
  • The system is shown, not just named. 29 reference cards, one concept each, a clickable UI kit, and 26 proposal mockups the audits compare against.
  • Brand rules are encoded as instructions. No emoji in product UI, Lucide stroke icons only, no gradients in chrome, sentence case for chrome and title case for product nouns, and a fixed decision-state vocabulary (none, pending, accepted, declined).
Same shape as the lessons library: institutional knowledge captured once as a versioned artifact the agent reads, rather than context re-supplied by hand every time.

The learning loop

Caught mistakes become rules the next run reads first

This is the part that makes it a system rather than a tool. When a project's last PR merges, the closeout ceremony mines every review finding for recurring families and writes them where the next agent looks before it codes. A mistake that recurs across two or more PRs becomes a rule.

↩ feedback: the next implementation starts by reading the lessons and the recurring-patterns checklist, so the same family does not return
Why a family, not a fix.

Fixing one bug helps one PR. Naming the family (its trigger, failure mode, and the check that prevents it) and putting it where the next agent reads first is what stops the class from coming back. The repo has 64 lesson files today. The single most-repeated finding, tenant-isolation tests that pass even when the tenant filter is dropped, still recurred four times across three PRs after it was documented. The response was to strengthen the trigger, not re-file the lesson.

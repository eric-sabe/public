## Core Data Structures

The application's core data structures intended for sharing between frontend and backend are primarily defined in `packages/shared-types/src/game-types.ts`. **Important:** These shared types are often simplified representations and may not include all fields or relations present in the backend's Prisma database models. Refer to `packages/backend/prisma/schema.prisma` for the definitive database structure.

Some types (like `GameEffectType` and `AssetType`) are defined both here and generated by Prisma; the backend typically uses the Prisma-generated versions.

##### ImmediateOptionalBuyAsset

Used for cards that force an immediate buy/pass decision upon being drawn (e.g., "Uncle Bert's Legacy").

**Difference from OptionalBuyAsset:**
- `ImmediateOptionalBuyAsset`: Decision required immediately, triggers modal dialog with 30-second timeout
- `OptionalBuyAsset`: Card held in hand, player can exercise later at will

**Effect Details Structure:**
```typescript
{
  asset: AssetType;      // Asset type to purchase (e.g., 'Hay')
  quantity: number;      // Quantity to purchase (e.g., 10)
  cost: number;         // Full cost required (e.g., 10000)
}
```

**Flow:**
- **Human Players**: Socket event triggers frontend dialog with countdown timer
- **AI Players**: Inline decision via profile-based logic
- **Timeout**: 30 seconds auto-pass if no decision made
- **Disconnect**: Auto-pass pending decisions when player disconnects

**AI Decision Factors:**
- Current asset portfolio (avoids over-concentration)
- Game phase (early/mid/late game strategy)
- Cost per unit analysis (value assessment)
- AI profile thresholds (Gambler vs Cautious vs Balanced)

### Game Phases

The game has four distinct phases (`GamePhase`), which correspond to the four sides of the game board:

- `SpringPlanting`
- `EarlySummer`
- `LateSummer`
- `EndOfYear`

The current phase is inferred dynamically based on the active player's board position (`PlayerPublic.position`) using a utility function (e.g., `getPhaseForTileIndex`) rather than being stored directly or mapped in a static config file.

### Temporary Harvest Modifiers

Represents temporary changes to a player's harvest yield, stored as JSON in the backend `Player` model. Defined in `shared-types` with an interface and a Zod schema for validation.

```typescript
/**
 * Represents a temporary harvest modifier for a player.
 * - `harvestType`: The crop/livestock affected (e.g., Corn, HayCutting1).
 * - `multiplier`: The yield multiplier (e.g., 2 for double, 0.5 for half, 0 for none).
 * - `expiresAfterYear`: How many years (turns) this modifier lasts (default 1, decremented at year end, removed if 0).
 * - `tileIndex`: (Optional) If present, restricts modifier to a specific board tile/zone.
 */
export interface TemporaryHarvestModifierEntry {
  harvestType: HarvestType;
  multiplier: number;
  expiresAfterYear?: number; // Defaults to 1 if not supplied
  tileIndex?: number;
}

export type TemporaryHarvestModifiers = TemporaryHarvestModifierEntry[];
```

## Database Schema

The backend database schema is defined using Prisma in `packages/backend/prisma/schema.prisma`.

**This file is the single source of truth for the database structure.** Please refer directly to `schema.prisma` for the most up-to-date and complete model definitions, fields, relations, attributes, and enums.

### Core Game Models

- **`Game`**: Represents a single game session, tracking status, turn, phase, players, decks, ridges, etc. Contains a `state` JSON blob for dynamic data. For session management, includes fields like `endedByHost`, `endedByHostAt`, `endedByHostUserId`, `disconnectGracePeriod`, and `inactivityTimeout`.

- **`Player`**: Represents a player within a game, storing their state (cash, debt, position), assets, cards, stats, relations, and temporary effects (`temporaryHarvestModifiers` JSON). For session management, includes fields like `connectionStatus`, `lastDisconnectTime`, `lastReconnectTime`, `isSuspended`, `lastActivityTime`, `isAFK`, `disconnectedDuringTurn`, and `autoPlayEnabled`.

- **`PlayerAsset`**: Joins `Player` and `AssetType`, storing quantity and cost basis for owned assets (Hay, Grain, Fruit, Cows, Tractor, Harvester).

- **`Ridge`**: Represents leasehold properties available in a game, potentially leased by a player.

- **`BoardTile`**: Defines the properties of each tile on the game board for a specific game instance.

### Card System Models

- **`CardDefinition`**: Templates for all cards in the game with effect specifications.

- **`PlayerCard`**: Represents an instance of a card owned by a player, including its location (Hand, Discard, ActiveEffect).

- **`GameDeck`**: Manages the draw and discard piles (arrays of `CardDefinition` IDs) for each deck type within a game.

### User Management Models

- **`User`**: Complete user accounts with OAuth integration, roles, permissions, game statistics, and privacy settings.

- **`Invitation`**: Email-based invitation system for user onboarding with status tracking.

- **`AccountActivity`**: Comprehensive audit trail for all user account actions.

### Administrative Models

- **`Experiment`**: A/B testing framework for feature flags and experimentation.

- **`ExperimentParticipation`**: Tracks user participation in active experiments.

### Logging & Monitoring Models

- **`GameLogEntry`**: Stores persistent game event logs for review and debugging.

- **`NetWorthSnapshot`**: Stores periodic snapshots of player net worth for historical tracking.

- **`UnifiedLogEntry`**: Centralized logging system for backend and frontend events.

### Database Enums

The schema includes comprehensive enum types for type safety and consistency:

- **Game-related**: `GameStatus`, `AssetType`, `HarvestType`, `GameEffectType`, `TileEffectType`, `AIProfile`, `DeckType`, `CardType`
- **User-related**: `UserRole`, `UserStatus`, `InvitationStatus`, `AccountAction`
- **System-related**: `LogSource`, `LogSeverity`, `ExperimentCategory`, `ExperimentRiskLevel`

## Financial Ledger System

The Financial Ledger system tracks all cash and debt changes for players throughout a game, providing detailed audit trails for debugging and analysis.

### FinancialLedgerEntry

Represents a single financial transaction entry in a player's ledger.

```typescript
export interface FinancialLedgerEntry {
  id: string;                    // Unique identifier for the ledger entry
  gameId: string;               // Game this entry belongs to
  playerId: string;             // Player this entry belongs to
  turn: number;                 // Turn number when the change occurred
  tile: number;                 // Tile position when the change occurred
  cash: number;                 // Cash amount after the change
  debt: number;                 // Debt amount after the change
  reason: string;               // Description of the transaction (e.g., "Game Start", "Tile effect: Gain $500")
  scoreboardMatch: boolean;     // Whether cash/debt values matched Player model at write time
  createdAt: string;            // ISO timestamp when entry was created
}
```

### Backend Implementation

**Database Model (`FinancialLedger`):**
- Stored in `financial_ledgers` table
- Foreign keys to `Game` and `Player` models
- Indexed on `(gameId, playerId)` and `(playerId, createdAt)` for efficient queries

**Service (`FinancialLedgerService`):**
- `logLedgerEntry()`: Creates new ledger entries with transaction context
- `getPlayerLedger()`: Retrieves all entries for a player, ordered by creation time

**Integration Points:**
- **Game Creation**: Logs initial cash/debt setup as "Game Start"
- **Tile Effects**: Logs payments, income, and forced loans
- **Option Purchases**: Logs asset/ridge purchases with cash/debt changes
- **Card Effects**: Logs Farmer's Fate and Operating Expense card effects
- **Loan Payments**: Logs debt reduction payments

### API Access

**Endpoint:** `GET /api/games/:gameId/players/:playerId/ledger`
- Returns array of `FinancialLedgerEntry` objects
- Requires authentication and game ownership verification
- Ordered by `createdAt` (oldest first)

### Frontend Integration

**Modal Component (`FinancialLedgerModal`):**
- Displays ledger entries in a sortable table
- Columns: Turn, Tile, Cash, Debt, Reason, Scoreboard Match, Timestamp
- Accessible by clicking Cash or Debt values in scoreboard
- Shows player name and handles loading/error states

**Scoreboard Integration:**
- Cash and Debt table cells are clickable (cursor: pointer)
- Hover tooltip: "Click to view financial ledger"
- Opens same modal for both Cash and Debt clicks
- Requires `gameId` prop to be passed to `StyledScoreboard`

### Usage Examples

**Viewing Ledger:**
1. In game scoreboard, click on any player's Cash or Debt amount
2. Modal opens showing chronological list of all financial changes
3. Green checkmark indicates values matched database at time of logging
4. Red X indicates potential synchronization issues for debugging

**Debugging Financial Issues:**
- Compare ledger entries against expected game events
- Use `scoreboardMatch` flag to identify data consistency issues
- Cross-reference with game logs for complete audit trail

### Data Flow

1. **Transaction Occurs**: Player cash/debt changes (tile effect, card play, etc.)
2. **Database Update**: Player model updated with new cash/debt values
3. **Ledger Entry**: `FinancialLedgerService.logLedgerEntry()` called with final values
4. **Sync Check**: Compares logged values against current Player model
5. **Storage**: Entry saved to `financial_ledgers` table with `scoreboardMatch` flag
6. **Display**: Available via API for frontend modal consumption

This system provides complete financial transparency and debugging capabilities for the farming game.
